# Защитное слово на 5-7 минут

Уважаемые члены комиссии, вашему вниманию представляется дипломный проект на тему: "Разработка MVP AI-агента для автоматизации B2B-процессов стоматологических клиник".

Цель моей работы заключалась в том, чтобы разработать рабочий MVP AI-агента, который помогает стоматологической клинике автоматизировать первичную административную коммуникацию с пациентами. Важно сразу обозначить границу проекта: это не медицинская экспертная система и не замена врачу. Агент работает как помощник администратора: принимает сообщения, понимает контекст, отвечает на типовые вопросы, собирает данные для записи, сохраняет историю и передает рискованные ситуации человеку.

Актуальность проекта связана с тем, что пациенты все чаще предпочитают писать в мессенджер, а не звонить. Для клиники это удобно, но создает нагрузку на администратора. Сообщения приходят в разное время, пациент может писать коротко, с ошибками, частями: например, "болит зуб", "можно сегодня", "сколько стоит кариес". При ручной обработке легко потерять контекст, забыть уточнить телефон, перепутать дату или назначить двух пациентов на одно время. Поэтому клинике нужен не просто чат-бот с кнопками, а система, которая умеет работать со свободным текстом и при этом сохраняет контроль над важными действиями.

В рамках проекта были поставлены следующие задачи: проанализировать процесс обработки обращений в стоматологической клинике, сформулировать требования к системе, спроектировать архитектуру, реализовать серверную часть, интеграцию с MAX, базу данных, административный интерфейс, логику записи, напоминания и механизмы безопасности.

Архитектура решения построена как гибридная система. Пациент общается через web-интерфейс MAX. MAX-адаптер на Node.js и Playwright открывает браузер, читает новые сообщения и отправляет ответы. n8n используется как workflow-оркестратор и связывает компоненты между собой. Основная бизнес-логика находится в Agent API на Express, а PostgreSQL хранит контакты, переписки, память диалога, заявки, слоты, напоминания и действия агента.

Такое разделение выбрано осознанно. Если бы вся логика была только в браузерной автоматизации, проект было бы сложно сопровождать и тестировать. Если бы все было только в n8n, было бы труднее реализовать сложные правила записи, проверки и безопасности. Поэтому MAX-адаптер отвечает за канал общения, n8n - за маршрутизацию, Agent API - за принятие решений, а PostgreSQL - за надежное хранение данных.

Сценарий обработки сообщения выглядит следующим образом. MAX-адаптер регулярно проверяет чаты и определяет, где последнее сообщение является входящим. Затем он формирует payload с идентификатором пользователя, именем, текстом и external message id. Через n8n сообщение попадает в Agent API. API находит или создает контакт и активную переписку, сохраняет входящее сообщение, загружает последние сообщения и память диалога, после чего запускает AI pipeline.

AI pipeline в проекте устроен не как свободная генерация текста. Модель используется для понимания естественного языка: она определяет намерение пациента, услугу, жалобу, дату, время, уровень срочности и предлагает структурированный результат. Но критичные действия выполняет не модель, а код. Модель может предложить action, например создать запись, но backend проверяет, есть ли все обязательные данные: услуга или жалоба, дата, время, контакт пациента и явное подтверждение. Если данных недостаточно, действие понижается до уточняющего вопроса.

Отдельное внимание уделено базе данных. В PostgreSQL хранятся contacts, conversations, messages, conversation_memory, appointment_requests, appointment_slots, appointment_reminders, handoffs и другие сущности. Благодаря этому переписка превращается в структурированную информацию. Ключевой пример - защита от двойной записи: таблица appointment_slots имеет уникальное ограничение по дате и времени. Даже если два пациента почти одновременно попросят один и тот же слот, база данных не позволит занять его дважды.

В проекте реализована память диалога. Это важно, потому что пациент редко сообщает все данные сразу. Он может сначала написать жалобу, потом имя, потом удобное время. Conversation memory хранит такие факты в структурированном виде, и агент не задает одни и те же вопросы повторно.

Для медицинской предметной области особенно важна безопасность ответов. Агент не ставит диагнозы, не назначает лечение, не указывает дозировки и не обещает результат. При тревожных симптомах, конфликте, угрозе плохого отзыва, юридической претензии или жалобе на неправильную запись система создает handoff и переводит диалог в режим участия администратора. Кроме того, перед отправкой в MAX ответ проходит sanitizer: пользователю уходит только чистый человеческий reply, без JSON, reasoning, action, memory или внутренних статусов.

Также реализована защита от дублей. Используется external_message_id, блокировка обработки переписки, проверка повторяющегося входящего текста и защита от повторной отправки одинакового исходящего ответа. Это важно для реального мессенджера, где сообщения могут быть обработаны повторно или watcher может сработать несколько раз.

Для администратора реализована web-панель. В ней можно смотреть статистику, переписки, заявки, handoff, напоминания, базу данных, промпты и базу знаний клиники. Это делает систему полезной не только для пациента, но и для сотрудников клиники: администратор видит, что уже произошло в диалоге, а руководитель получает общую картину по обращениям.

Кроме записи реализованы напоминания и outbound-сценарии. После подтверждения записи создаются напоминания о визите, а плановые сообщения отправляются только в разрешенное дневное окно. Также предусмотрены рассылки и повторные касания, например напоминание пациентам о профессиональной гигиене через заданный период.

Работоспособность проверялась не только отдельными ответами, а полным циклом: сообщение из MAX, обработка через n8n и Agent API, сохранение в PostgreSQL, создание заявки, проверка слота, формирование ответа, отправка в MAX и отображение в административной панели. Дополнительно предусмотрены тестовые сценарии для записи, вопросов по цене, отмены, срочных симптомов, грубости, мусорного текста, дублей и утечки JSON.

В результате был разработан рабочий MVP, который демонстрирует полный путь от сообщения пациента до сохраненной заявки и напоминания. Практическая значимость проекта заключается в снижении нагрузки на администратора, ускорении ответа пациенту, сохранении истории, уменьшении риска ошибок записи и подготовке управленческой информации.

При этом проект честно имеет ограничения MVP. MAX-адаптер зависит от web-интерфейса и селекторов, требуется дальнейшее промышленное размещение, расширение авторизации, мониторинг и интеграция с реальным расписанием врачей или медицинской информационной системой. Но архитектура уже подготовлена к развитию: можно добавить CRM/МИС, расширенную аналитику, кабинет руководителя, AI-ассистента по внутренним вопросам и сценарии повторной лидогенерации.

Таким образом, поставленная цель достигнута: разработан MVP AI-агента для стоматологической клиники, который принимает обращения через MAX, понимает контекст, безопасно отвечает, собирает данные для записи, предотвращает двойное бронирование, создает напоминания и предоставляет администратору панель контроля.
